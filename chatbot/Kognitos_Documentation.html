1. System Overview

The Kognitos AI Assistant is a full-stack automation interface that allows users to query complex enterprise data using plain natural language. The system connects a premium Chat UI to the Kognitos Automation Engine through a secure, serverless proxy layer.

High-Level Architecture
👤 User (Browser)
Types natural language query → Premium Chat UI in browser
↓ HTTP Request (No credentials exposed)
🔒 Vercel Backend Proxy
api/index.js  |  Injects API_TOKEN from environment  |  Formats Kognitos payload
↓ Authenticated HTTPS to Kognitos Cloud
🧠 Kognitos Engine
app.us-1.kognitos.com  |  Runs SOP Automation  |  Fetches cross-database data

public/style.css
All visual design: dark/light theme variables, pipeline tracker styles, result card layouts, and animations.
public/app.js
Core frontend logic: event handling, API polling, real-time activity tracker, and the complete result-rendering engine.
api/index.js
Vercel Serverless Function. Acts as the secure proxy between the browser and the Kognitos API.
server.js
Legacy local development server. Serves static files and proxies API requests when running locally with node server.js.
vercel.json
Vercel configuration. Defines routing rules: API paths → serverless function, everything else → static public files.
.env
NEVER commit this file. Contains secret credentials: ORGANIZATION_ID, WORKSPACE_ID, AUTOMATION_ID, API_TOKEN.
.gitignore
Ensures .env, node_modules, and log files are excluded from GitHub version control.

3. Request Lifecycle: Step-by-Step

The following is the exact sequence of events that occurs every time a user enters a query and presses Enter.

1
User Input → handleSend() You type "share the list of unregistered clients" and hit Enter. The handleSend() function in app.js fires. The UI immediately shows a thinking indicator and the "AI Execution Pipeline" tracker opens.
2
Frontend → Proxy: POST /api/invoke The browser sends a simple JSON payload to your own backend. No Kognitos credentials are in this request — they live safely on the server side only.
3
Proxy → Kognitos: Authenticated Invocation The backend (api/index.js) restructures the query into Kognitos' exact required format, attaches your Authorization: Bearer API_TOKEN header, and forwards it securely to app.us-1.kognitos.com.
4
Kognitos Responds with Run ID Kognitos confirms it has received the job and begins execution. It immediately returns a unique Run ID (e.g., xyz123). This ID is sent back to the browser.
5
Real-Time Event Streaming: /api/runs/xyz123/events Using the Run ID, the frontend starts a polling loop against the /events endpoint. As Kognitos runs its SOP, it logs execution steps (intent classification, DB fetch, field mapping). The frontend reads each log and updates the "AI Execution Pipeline" progress bar and terminal display in real-time.
6
Status Polling: /api/runs/xyz123 (every 2 seconds) Simultaneously, the frontend polls the run status endpoint every 2 seconds. This is lighter-weight and returns either "running" or "completed."
7
Completion Detected → Data Extraction When the status returns "completed," the response includes the full outputs object from Kognitos. This contains all the raw data (tables, metrics, response text) exactly as Kognitos produced it.
8
Data Processing: flattenKognitosTable() Because Kognitos returns deeply nested JSON (lists within dictionaries within lists), the flattenKognitosTable() function recursively unwraps all nesting into a clean, flat 2D array — rows and columns ready for rendering.
9
Premium UI Rendering The "AI Execution Pipeline" tracker collapses, turns green and shows "Pipeline Success." The final data is rendered as a premium, fully scrollable HTML table with metric cards above it. Every single row and field from Kognitos is displayed — nothing is filtered or distincted by the frontend.

4. API Endpoint Reference

Endpoint Method Purpose Kognitos Target
/api/invoke POST Triggers the Kognitos Automation SOP. Accepts the user's natural language query and returns a Run ID. .../automations/:id:invoke
/api/runs/:id GET Polls the current execution status. Returns "running" or "completed" along with final outputs on success. .../runs/:runId
/api/runs/:id/events GET Streams real-time execution journal logs. Used to power the live "AI Execution Pipeline" tracker in the UI. .../runs/:runId/events

5. Security Architecture

Protecting your Kognitos API_TOKEN is critical. Placing it in frontend JavaScript would expose it to anyone who opens the browser's "Inspect Element" panel.

⚠️ Never Put Credentials in Frontend Code

If API_TOKEN is placed in public/app.js or index.html, any visitor to your site can view it, steal it, and make unlimited API calls at your expense.

✅ How This Application Is Protected

All credentials (API_TOKEN, ORGANIZATION_ID, etc.) are stored server-side only as environment variables in Vercel. The browser never sees them. The browser only calls /api/invoke, which is your own proxy, which then attaches the key on the backend before forwarding to Kognitos.

Environment Variables Reference

VariableWhere It's UsedWhere to Set for Production
API_TOKEN Injected as Bearer token in all Kognitos requests. Vercel Dashboard → Settings → Environment Variables
ORGANIZATION_ID Identifies your Kognitos organization. Vercel Dashboard → Settings → Environment Variables
WORKSPACE_ID Identifies the specific Kognitos Workspace. Vercel Dashboard → Settings → Environment Variables
AUTOMATION_ID Points to the specific published SOP Automation. Vercel Dashboard → Settings → Environment Variables

6. Deployment & CI/CD Workflow

The project uses a zero-friction, automated deployment pipeline via GitHub and Vercel.

1
Local Development Make code changes. Run node server.js locally to test on http://localhost:3000.
2
Git Push to GitHub Run git add . → git commit -m "update" → git push. Code is now updated on github.com/Padmanaban-Dev/kognitos-chatbot.
3
Vercel Auto-Deploy (~30 seconds) Vercel detects the new code on GitHub. It automatically builds and deploys the updated version. Your live URL is instantly updated with no manual action required.
💡 Pro Tip: Using Vercel Preview Deployments

If you push to a new Git branch (e.g., feature/new-table-style), Vercel creates a separate temporary preview URL for it — allowing you to safely test a new feature without touching your main live site.
